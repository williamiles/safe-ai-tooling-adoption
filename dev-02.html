Prompt injection is a production problem, not a research problem.

It's been demonstrated in Microsoft 365 Copilot, GitHub, Slack, and every major platform that connects an LLM to private data. Here's the mechanism and the architectural test.

See the mechanism

The lethal trifecta

Three capabilities that are individually useful and collectively dangerous. When all three are present in a single agent system, data exfiltration via prompt injection becomes architecturally possible. Tap each leg to activate.

Lethal Trifecta
🔒
Access to private data
The agent can read emails, documents, databases, or other sensitive resources
📨
Exposure to untrusted content
The agent processes inputs it doesn't control: shared docs, emails, web pages, RAG context
🚀
External communication capability
The agent can send emails, post messages, make HTTP requests, or write to external systems

Production incidents — not proofs of concept

These happened in shipped products with real users. Tap each to see the mechanism.

M365 Copilot Data exfiltration via indirect prompt injection +
EchoLeak (CVE-2025-32711): a zero-click attack where a crafted email causes Microsoft 365 Copilot to extract and transmit private data without any user interaction. The attacker sends an email with hidden instructions phrased to avoid Copilot's prompt injection classifiers. When Copilot reads the email as context, it follows the injected instructions and exfiltrates sensitive data via Markdown reference-style links, a syntax Microsoft's filters didn't cover. The exfiltration exploited an open redirect on *.teams.microsoft.com to bypass Content Security Policy. Vector: crafted email with indirect prompt injection → Copilot reads as context → exfiltrates via Markdown reference links through Teams open redirect to attacker server
GitHub MCP Private repo exfiltration via poisoned issues +
Invariant Labs demonstrated that a malicious GitHub Issue in a public repository could hijack an AI agent into exfiltrating private repository data. The attacker creates an issue with embedded prompt injection instructions. When a developer's agent reviews issues in that public repo, it encounters the crafted issue, follows the injected instructions, accesses the user's private repositories, and leaks sensitive data, including private repo names, proprietary code, and personal information, via a pull request back to the attacker's public repo. Even Claude Opus's safety training was insufficient to prevent the attack. Vector: poisoned GitHub Issue in public repo → agent reads as context → accesses private repos via MCP → exfiltrates data via pull request to attacker-controlled repo
Slack AI Credential theft via indirect prompt injection +
PromptArmor demonstrated that an attacker could steal API keys, credentials, and private messages from Slack AI without access to private channels. The attacker posts a message in a public channel containing hidden instructions. When a user queries Slack AI, it pulls context from public channels, private channels, and DMs, treating all content with equal weight. The injected instructions cause Slack AI to render a fake error message with a "reauthentication link", embedding the stolen data as URL parameters. Slack's citation system failed to properly flag the attacker's content, making the source difficult to trace. Vector: injected message in public channel → Slack AI ingests alongside private data → renders phishing link with stolen credentials embedded as URL parameters

Why prompt injection is an architectural problem

Prompt injection is an attacker embedding instructions in content the AI reads, often without the user's knowledge or consent. The user never types anything malicious. The attack lives in an email, a shared document, a GitHub issue, a Slack message. Content the AI ingests as context.

This is not a guardrail problem. It's an architectural problem. LLMs cannot reliably distinguish between instructions and data in their context window. Every input is processed the same way.

No amount of prompt engineering, instruction tuning, or guardrail layering provides 100% protection against indirect prompt injection. In an adversarial context, 99% protection is a failing grade. The attacker only needs one successful injection.
