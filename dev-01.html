The 10 recurring classes

Every one of these is a well-known vulnerability class. Every one was introduced by AI agents in 2026. Tap to see the pattern.

#01
Broken access control
Missing authorization checks on state-changing endpoints. Agents generate functional routes but skip ownership verification. Any authenticated user can modify any resource.
#02
Business logic failures
Authorization logic that checks the wrong field or validates presence instead of correctness. The code runs, the tests pass, but the security invariant doesn't hold.
#03
OAuth implementation flaws
Missing state parameter in OAuth flows, enabling CSRF-based account takeover. Agents implement the happy path but omit anti-forgery protections.
#04
WebSocket auth gaps
WebSocket connections established without authentication checks. Agents set up the transport layer but skip the auth handshake, so any client can connect and receive events.
#05
Rate limiting absence
No rate limiting on authentication endpoints. Login, registration, and password reset routes are fully functional and fully brute-forceable.
#06
Weak JWT secrets
Hardcoded JWT secrets like "secret" or "your-secret-key". Agents generate working token flows but use placeholder secrets that ship to production.
#07
2FA bypass
2FA implementation that can be disabled without re-authentication. One agent introduced a 2FA-disable endpoint that accepted any authenticated request. No password confirmation, no existing 2FA challenge.
#08
User enumeration
Login and registration endpoints that return different error messages for "user not found" vs "wrong password." Agents optimize for helpful UX and leak account existence.
#09
Session management failures
Sessions not invalidated on password change or logout. Old tokens remain valid indefinitely. A compromised session stays compromised.
#10
Non-revocable refresh tokens
Refresh tokens with no revocation mechanism. Once issued, they can't be invalidated, even if the user changes their password or their account is compromised.

Case study: Moltbook

What happens when these vulnerability classes ship to production at platform scale.

Production incident Moltbook — AI social network
Moltbook was built rapidly using AI development tools, the kind of vibe-coded, ship-fast approach that's increasingly common. Wiz Research found the platform had no Row Level Security (RLS) policies on its Supabase backend. The exposed database was accessible without authentication.
RLS No Row Level Security policies configured. Full database access without authentication
TOKENS 1.5M authentication tokens for AI agents exposed publicly
PII 35,000+ email addresses and private messages containing third-party API keys (including OpenAI keys)
WRITE Write access enabled prompt injection and content integrity attacks beyond data exposure
RATIO 88:1 agent-to-human ratio (1.5M agents, 17K human owners), no verification mechanisms

What to change in your workflow

Specific process changes. Tap to commit.

Scan every AI-generated PR individually, not just full-repo scans
Add security requirements to your prompts: auth checks, rate limiting, secret rotation
Flag JWT/session/auth PRs for mandatory security review. They cluster the most vulnerabilities
Use contextual analysis tools that trace data flows, not just pattern-matching scanners
Conduct security review during the planning phase, before the agent generates code
Write a security test first, verify it fails, then implement the fix and confirm it passes
Add the 10 vulnerability classes to your PR review template as explicit checkpoints
