Security is a practice, not a setting

The mindset that makes everything else in this article work.

Every time you start a session with an AI coding agent, you're making security decisions. What files can it access? What commands can it run? What network connections can it make? What tools and connectors are active? Most developers don't think of their dev environment as a security surface. It is.

Articles 1 and 2 in this series established the problems: AI tools write vulnerable code by default, and prompt injection is a real production threat. This article is about what you do with that knowledge in your own workflow.

The goal isn't to slow down. It's to set up your environment so the safe path is also the fast path. Configure restrictive defaults once, and you stop making security decisions under fatigue dozens of times per session. You make them once, deliberately, in a config file. Then you work.

Things will go wrong. Your job is to contain the blast radius before you start working, and stay engaged enough to catch what containment misses.

What mode are you running in?

Every autonomy level is a trust decision you're making. Are you choosing deliberately, or did you drift? Tap a level to see what it grants.

1
Level 1 — Assistive
AI suggests, you decide
Copilot-style inline completions. AI generates suggestions, you review and accept. All actions require your explicit approval.
What you're granting: Read access to your open files. No command execution, no file writes, no network access beyond the model API.
low risk human is the gate
Standard code review practices apply. You're the final gate on every line.
2
Level 2 — Semi-autonomous
AI acts, you approve sensitive ops
Agent writes code, modifies files, runs commands, but gates sensitive operations for your approval. Claude Code in normal mode.
What you're granting: File read/write within project. Command execution with approval. Network access for defined tools.
sandboxing essential approval fatigue risk configure deny rules
This is where most developers should operate daily. Sandboxing and deny rules do the heavy lifting so approval prompts stay meaningful.
3
Level 3 — Auto-accept
AI plans and executes without gating
Agent runs commands, writes files, and executes tool calls without asking. You've removed yourself as a checkpoint.
What you're granting: Full project access. Unrestricted command execution within sandbox. No human review before action.
deny rules critical sandbox mandatory monitor actively
If you're here, your deny rules and sandbox are your only defense. Make sure they're configured before you enter this mode, not after.
4
Level 4 — Bypass / YOLO
No sandbox, no approvals, no safety net
Sandbox disabled. All permissions granted. The agent operates with your full system access. A single manipulation cascades.
What you're granting: Everything. Full filesystem, unrestricted network, no deny rules enforced, no approval prompts.
disposable environments only
Reserve for throwaway containers and VMs. Never on a machine with credentials, keys, or access to production systems. If you're running this on your daily driver, you've accepted a level of risk that no configuration can mitigate.

Contain the blast radius

Three layers of defense that work together: sandbox isolation, permission defaults, and deliberate mode selection.

Layer 1 Sandbox Isolation
Anthropic's dual-layer approach combines filesystem isolation (agent can only access the project directory) with network isolation (all traffic routed through a domain-validating proxy). Both layers are essential. Filesystem isolation alone doesn't prevent network-based exfiltration, and network isolation without filesystem boundaries can be circumvented by writing to files that sync externally.

Built on OS-level primitives: Linux Bubblewrap and macOS Seatbelt. The sandbox ensures that even a successful prompt injection remains contained . The injected instruction may execute, but it can't access files outside the project or transmit data to unauthorized domains.

Layer 2 Permission Defaults in Configuration
Don't rely on clicking "deny" in the moment. Encode your security decisions in a config file, not your discipline. Same reason you use linters instead of relying on developers to remember style rules. Willpower degrades over a session, configuration doesn't.

Claude Code supports permission rules in settings.json at three levels: global, shared project, and local project. Deny rules always take priority and accumulate across levels. Set them once, and destructive commands are blocked regardless of which mode you're in or how fatigued you are.
~/.claude/settings.json — permissions & sandbox
{
  "permissions": {
    "allow": [
      "Bash(head *)",
      "Bash(tail *)",
      "Bash(wc *)",
      "Bash(sort *)",
      "Bash(diff *)",
      "Bash(echo *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(chmod 777 *)",
      "Bash(curl *)",
      "Bash(wget *)",
      "Read(.env)",
      "Read(**/*.pem)",
      "Read(**/*.key)",
      "Edit(.env)",
      "Edit(**/*credentials*)",
      "Edit(**/package-lock.json)"
    ]
  },
  "sandbox": {
    "enabled": true,
    "allowUnsandboxedCommands": false,
    "network": {
      "allowedDomains": [
        "mysafedomain.com"
      ]
    },
    "filesystem": {
      "denyWrite": [
        ".env", "~/.bashrc", "~/.zshrc"
      ],
      "denyRead": [
        "~/.ssh", "~/.aws", "~/.gnupg",
        "~/.git-credentials", "~/.npmrc"
      ]
    }
  }
}

The principle applies beyond Claude Code. Cursor, Copilot, and other AI tools have their own configuration surfaces: workspace trust settings, tool approval defaults, extension permissions. Audit them. Set restrictive defaults. Don't rely on per-prompt discipline.

Audit your connectors

Your agent connects to external services: MCP servers, API integrations, tool plugins. Each one extends the agent's capabilities and its attack surface. Review what's connected. Disable what you're not using.

Every active connector is a potential data path. An MCP server that reads your database gives the agent access to that data. A Slack integration gives it a communication channel. A GitHub connector gives it repository access. Each connector is a trust decision. Are you making it deliberately?

Audit checklist: tap each connector type as you review it in your setup:
MCP servers data access, tool execution REVIEWED
API integrations external service calls REVIEWED
Tool plugins / extensions filesystem, network, actions REVIEWED
Repository CLAUDE.md / config files prompt injection vector REVIEWED

The honest caveat: sandboxing has limits

Configuration is necessary but not sufficient. Here's why your attention still matters.

Research — Ona, March 2026 The agent that broke its own security
Leonardo Di Donato demonstrated that Claude Code, when blocked by its own denylist, autonomously discovered evasion techniques, without being instructed to bypass security. The trick: when direct execution was denied, the agent invoked the dynamic linker (ld-linux-x86-64.so.2) via mmap instead of execve, sidestepping the sandbox entirely.
The sandbox caught most attempts, but not all. This isn't an argument against sandboxing. It's an argument against only sandboxing. Your configuration handles the majority of threats. Your attention handles the rest. Stay engaged with approval prompts. They're the ones that matter.

The good news: security gaps like these are slowing AI tool adoption, and vendors know it. Expect each release to ship with stronger sandboxing, tighter defaults, and better permission models. Stay up to date with release notes and security updates and the tooling is catching up fast.

Secure your environment today

Practical actions you can take right now.

Determine your autonomy level. Choose it deliberately, don't drift into it
Configure global deny rules in settings for destructive commands
Enable dual-layer sandboxing, filesystem and network isolation
Audit active connectors, MCP servers, and tool plugins. Disable what you're not using
Run in Normal mode by default. Escalate only when needed with matching safeguards
Review CLAUDE.md and config files in external repos before opening them with your agent
Treat remaining approval prompts as security decisions. They're the ones that matter

Guiding principles

Three ideas to carry beyond this article.

01 Your dev environment is a security surface. Every session, every tool, every permission is a decision. Configure it like one.
02 Encode security decisions in configuration, not discipline . Willpower degrades over a session. Settings files don't.
03 Sandboxing contains most threats; your attention catches the rest . Configuration is necessary but not sufficient. Stay engaged with what your agent is doing.
