From your environment to your architecture

A different security surface: the systems you ship to users.

Article 3 was about securing the tools you use to build. This article is about securing what you build with them: AI-integrated applications, agent systems, RAG pipelines, and any product where an LLM processes untrusted input or accesses private data.

The lethal trifecta from Article 2 (private data access + untrusted content + external communication) is the threat model. Every pattern here architecturally eliminates or constrains at least one leg. They're complementary. Layer them. And they're paired with a framework for thinking about agent security from first principles: what to evaluate, what to enforce, and what to observe.

Dual LLM Architecture

Simon Willison / independent research
Separate the LLM that processes untrusted content from the LLM that has access to private data and tools. The "privileged" LLM never sees raw untrusted input. The "quarantined" LLM processes external content and produces structured outputs that are validated before passing to the privileged layer. An injected instruction in untrusted content reaches the quarantined LLM, which has no tools to exploit.
How it works

Untrusted input → quarantined LLM (no tools, no private data) → structured output → validator → privileged LLM (tools, data)

Limitations

Adds latency and complexity. Requires clear separation of trusted vs untrusted context. Validator must be robust against structured output manipulation.

Trifecta leg eliminated: Breaks the link between untrusted content and private data/tools

Dual-Layer Sandboxing

Anthropic — Claude Code, October 2025
Combine filesystem isolation and network isolation using OS-level primitives. Filesystem isolation restricts which files the agent can access. Network isolation routes all traffic through a proxy that validates domains. Both layers are essential. Filesystem alone doesn't prevent network exfiltration, and network alone can be circumvented through filesystem manipulation. Article 3 covered this for your dev environment; here it applies to agent systems you deploy for users.
How it works

Agent runs in sandboxed process → filesystem access restricted to scoped directory → network traffic routed through domain-validating proxy

Limitations

Requires OS-level integration. Agent functionality may be constrained. Allowlist maintenance for legitimate domains is ongoing work. Path-based enforcement has known bypass vectors (see Article 3).

Trifecta legs constrained: Limits private data access (filesystem) + eliminates exfiltration (network)

Context Minimization

Google SAIF / defense-in-depth principle
The agent should only see the data it needs for the current task, nothing more. Instead of giving an agent read access to an entire email inbox, retrieve only the specific email being processed. Instead of passing full database contents into RAG context, return only the relevant rows. This reduces the blast radius of a successful injection: even if the agent is manipulated, the data available to exfiltrate is minimal.
How it works

Task-scoped data retrieval → agent receives minimal context → injected instructions can only access what's in the current context window

Limitations

Requires careful scoping of data retrieval. May reduce agent effectiveness for tasks that genuinely need broad context. Implementation varies by data source.

Trifecta leg constrained: Minimizes private data exposure. Limits the blast radius of successful injection

Network Allowlisting

Anthropic sandbox / zero-trust networking
Route all agent network traffic through a proxy that only allows connections to a pre-approved list of domains. An injected instruction telling the agent to send data to attacker.com fails because the proxy blocks the connection. This is the most direct way to eliminate the exfiltration leg of the trifecta. The agent physically cannot transmit data to unauthorized endpoints.
How it works

Agent → network proxy → domain validated against allowlist → approved: connection proceeds / denied: connection blocked + logged

Limitations

Allowlist must be maintained as legitimate integrations change. Doesn't prevent exfiltration through approved domains (e.g., embedding data in approved API calls).

Trifecta leg eliminated: Removes the exfiltration vector, the hardest leg to bypass when implemented at the network layer

Human-in-the-Loop Gating

Google SAIF / NVIDIA autonomy levels
Require explicit human approval for sensitive operations: sending external communications, accessing restricted data, executing destructive actions. The gate must be on the action, not the plan. An agent can plan to exfiltrate data freely. The defense activates when it attempts to execute the action. Effective at lower autonomy levels but degrades at higher levels due to approval fatigue.
How it works

Agent plans action → action classified as sensitive → human reviews and approves/denies → only approved actions execute

Limitations

Approval fatigue leads to rubber-stamping. Doesn't scale to high-autonomy agents. Must be paired with sandboxing to reduce prompt volume to a sustainable level.

Trifecta leg constrained: Gates external communication, but only as reliable as the human reviewing it

Hard-Coded Guardrails

SANS Institute — zero-trust AI principle
Define actions that the AI can never perform, regardless of instructions, context, or confidence level. These are not probabilistic defenses. They're deterministic code checks that execute before the AI's output reaches any system. if action == "delete_database": deny(). No LLM reasoning, no prompt engineering, no guardrail that can be talked around. Hard-coded in application logic, not in the AI's instruction set.
How it works

AI output → deterministic policy check (not LLM-based) → allowed actions proceed / prohibited actions blocked unconditionally

Limitations

Only works for enumerable, predictable actions. Can't guard against novel or creative exfiltration methods. Must be maintained as the action space evolves.

Trifecta leg constrained: Prevents specific dangerous actions. Deterministic, not probabilistic

Evaluate — Assess before acting

Before an agent takes any action, evaluate the risk. This includes classifying the action type (read, write, execute, communicate), checking it against policy, and assessing whether the current context warrants the requested operation. Evaluation happens at the application layer, not inside the LLM's reasoning.
classify_action(tool_call) → { type: "write", target: "production_db", risk: "high" }
check_policy(action, user_context) → DENY | APPROVE | REQUIRE_REVIEW
validate_parameters(tool_call) → reject if params exceed scoped permissions

Enforce — Constrain at runtime

Apply constraints that the agent cannot bypass, regardless of its instructions. Hard-coded guardrails, sandboxed execution, network allowlists, and filesystem restrictions. Enforcement must be deterministic and implemented outside the LLM: in application code, at the OS level, or in network infrastructure.
sandbox.restrict(fs: ["/project"], net: ["api.internal.com"])
if action.type == "delete" && action.target.is_production: BLOCK
proxy.allow_domains(["github.com", "npm.org"]) // deny all others

Observe — Detect after the fact

Log every action, monitor for anomalies, and maintain audit trails. Even with evaluation and enforcement, novel attacks will eventually succeed. Observation ensures you detect them, understand the blast radius, and can trace the full chain of actions. Every agent action must be attributable to a specific identity and context. No anonymous operations.
audit_log.append({ agent, action, target, timestamp, user_context })
alert_if(network_requests.to_unknown_domains > 0)
trace(agent_session) → full action chain with identity propagation

Trust boundaries for agent systems

Every input, output, and tool call crosses a trust boundary. Map yours explicitly.

Trusted zone — your code, your infra
system prompt tool definitions policy engine guardrails audit logs
── TRUST BOUNDARY ──
Semi-trusted — authenticated but not controlled
user input internal APIs database content authenticated services
── TRUST BOUNDARY ──
Untrusted — assume adversarial
RAG context shared documents email content web pages third-party APIs user-uploaded files
The rule: Data from untrusted zones must never flow into trusted zones without validation. The agent must never execute instructions originating from untrusted content. Identity must propagate through every boundary crossing. No action is anonymous.

Data poisoning: the supply chain attack for AI

If you fine-tune models or use RLHF pipelines, this is the threat that bypasses all runtime defenses.

Research — Anthropic, Oct 2025 Small Samples, Big Poison
Anthropic demonstrated that as few as 250 poisoned documents can create backdoors across all tested model sizes (600M to 13B parameters). The key finding: attackers need a fixed absolute number of poisoned samples, not a percentage of training data, making the attack trivially cheap regardless of dataset scale.
COST 250 documents reliably succeeded, a near-constant number regardless of model or dataset size
TRIGGER Specific strings activate the backdoor; outputs appear normal otherwise
DEFENSE Provenance tracking and contributor trust scoring for all training data sources
